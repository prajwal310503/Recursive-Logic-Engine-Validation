I went back and forth on whether to use a nested structure or a normalised one. The nested approach feels natural at first — each node just holds its children directly, like a tree. But the moment I wanted users to create cross-links between arbitrary nodes, that broke down. The same node would have to live in multiple places, or I'd need to pass around pointer-style references that make React updates a nightmare. Every state change would mean cloning an entire ancestry chain just to update one field.

So I went with a flat normalised map instead — every node is stored by its ID at the top level, and relationships are just IDs pointing at each other. Looking up any node is O(1). Updating a node means spreading one object. Cross-links are trivial — just store an ID. And crucially, cycle detection becomes a plain graph traversal rather than trying to navigate a tangle of nested references.

For cycle detection I used an iterative DFS with three-colour marking — white for unvisited, gray for currently on the stack, black for fully processed. The key insight is that if you're walking the graph and you hit a gray node, you've found a back edge — something is pointing back up into your current path, which is a cycle by definition. I track the path explicitly so I can highlight exactly which nodes are part of the loop, not just flag that one exists.

I made the traversal iterative rather than recursive because JavaScript's call stack isn't deep enough for arbitrarily nested trees, and the app lets users build trees of any depth. It also runs on every structural change — add a child, draw a link, delete a node — so keeping it O(V + E) felt important. The algorithm checks both tree edges and cross-link edges in the same pass, so nothing slips through.
